# Clarify — 스펙 문서 모호성 분석 스킬

`/clarify` 뒤에 전달된 입력으로 대상 피처 스펙을 찾아 분석한다.

## 실행 흐름

### 1. 스펙 파일 탐색

`/clarify` 뒤의 인자를 파싱하여 `specs/` 하위에서 대상 스펙을 찾는다.

- 숫자만 전달된 경우 (예: `02`): `specs/` 에서 `02_` 로 시작하는 디렉토리를 찾는다
- 디렉토리명이 전달된 경우 (예: `02_room-and-lobby-api-mvp`): 해당 디렉토리를 직접 사용
- 매칭되는 디렉토리가 없으면: "해당 번호/이름의 스펙 디렉토리를 찾을 수 없습니다. `specs/` 하위 디렉토리를 확인하세요." 출력 후 종료
- `spec.md` 파일이 없으면: "/specify를 먼저 실행하여 스펙을 생성하세요." 출력 후 종료

### 2. 스펙 분석 — 모호성 & 커버리지 스캔

스펙 파일을 읽고, 아래 분류 체계에 따라 각 카테고리의 상태를 내부적으로 판정한다 (Clear / Partial / Missing). 이 판정 결과는 질문 생성 여부 결정과 질문 도출에 사용하며, 사용자에게 직접 출력하지 않는다.

#### 분류 체계

**기능 범위 & 행동 (Functional Scope & Behavior)**
- 핵심 사용자 목표 & 성공 기준
- 명시적 범위 외 선언
- 사용자 역할/페르소나 구분

**도메인 & 데이터 모델 (Domain & Data Model)**
- 엔티티, 속성, 관계
- 식별 & 고유성 규칙
- 생명주기/상태 전이
- 데이터 규모/스케일 가정

**인터랙션 & UX 흐름 (Interaction & UX Flow)**
- 핵심 사용자 여정/시퀀스
- 에러/빈 상태/로딩 상태
- 접근성 또는 현지화 관련 사항

**비기능 품질 속성 (Non-Functional Quality Attributes)**
- 성능 (지연시간, 처리량 목표)
- 확장성 (수평/수직, 한계)
- 신뢰성 & 가용성 (업타임, 복구 기대치)
- 관측성 (로깅, 메트릭, 트레이싱)
- 보안 & 프라이버시 (인증/인가, 데이터 보호, 위협 가정)
- 규정 준수/규제 제약

**통합 & 외부 의존성 (Integration & External Dependencies)**
- 외부 서비스/API 및 장애 모드
- 데이터 임포트/익스포트 형식
- 프로토콜/버저닝 가정

**엣지 케이스 & 장애 처리 (Edge Cases & Failure Handling)**
- 부정적 시나리오
- 속도 제한/스로틀링
- 충돌 해결 (예: 동시 편집)

**제약조건 & 트레이드오프 (Constraints & Tradeoffs)**
- 기술적 제약 (언어, 스토리지, 호스팅)
- 명시적 트레이드오프 또는 거부된 대안

**용어 & 일관성 (Terminology & Consistency)**
- 표준 용어집 용어
- 회피할 동의어/폐기된 용어

**완료 신호 (Completion Signals)**
- 인수 기준의 테스트 가능성
- 측정 가능한 완료 정의 지표

**기타 (Misc)**
- TODO 마커/미해결 결정 사항
- 정량화 없는 모호한 형용사 ("견고한", "직관적인" 등)

### 3. 질문 생성

Partial 또는 Missing 상태인 카테고리에서 명확화 질문 후보를 생성한다. 다음 기준으로 필터링한다:

**포함 기준** — 답변이 다음 중 하나에 실질적으로 영향을 미치는 질문만:
- 아키텍처 또는 데이터 모델링
- 태스크 분해
- 테스트 설계
- UX 행동
- 운영 준비성
- 규정 준수 검증

**제외 기준**:
- 스펙에서 이미 답변된 내용
- 사소한 스타일 선호
- 구현 계획 단계에서 결정할 세부사항 (기능적 정확성을 차단하지 않는 한)
- 명확화해도 구현이나 검증 전략이 실질적으로 변하지 않는 항목

포함/제외 기준을 통과한 질문만 생성한다. 수량 상한선은 없다 — 기준을 통과하는 질문이 1개면 1개, 7개면 7개를 생성한다. 기준에 맞지 않는 질문을 억지로 채우지 않는다.

유효한 질문이 없으면 즉시 "명확화가 필요한 중요한 모호성이 발견되지 않았습니다."라고 보고하고 종료한다.

### 4. clarification.md 생성

스펙 파일과 같은 디렉토리에 `clarification.md`를 생성한다. 형식은 다음과 같다:

```markdown
# Clarification: [피처 이름]

> 생성일: YYYY-MM-DD
> 대상 스펙: spec.md
> 총 질문 수: N

---

## Q1: [질문 제목]

**카테고리**: [분류 체계 카테고리명]

**맥락**: [스펙에서 이 질문이 발생한 배경 — 관련 섹션 인용 또는 요약]

**질문**: [구체적이고 명확한 질문]

**답변**:

---

## Q2: [질문 제목]

...
```

각 질문의 구성:
- **카테고리**: 어떤 분류에서 도출된 질문인지
- **맥락**: 스펙의 어떤 부분에서 모호성이 발견되었는지 (직접 인용 또는 요약)
- **질문**: 하나의 명확한 질문. 짧은 답변(5단어 이내)이나 선택지로 답할 수 있도록 구성
- **답변**: 빈칸 (이후 별도 단계에서 채움)

### 5. 완료 보고

생성 완료 후 다음을 출력한다:
- `clarification.md` 파일 경로
- 생성된 질문 수
- 질문이 없는 경우 그 사실과 다음 단계 제안

## 행동 규칙

- **분석 대상은 오직 spec.md 파일뿐이다** — 백엔드 코드, API 구현, 데이터베이스 스키마 등 코드베이스의 다른 파일을 탐색하거나 참조하지 않는다. 이 스킬의 목적은 스펙 문서 자체의 모호성을 찾는 것이지, 구현과 스펙 사이의 차이를 찾는 것이 아니다.
- 스펙 파일이 없으면 `/specify`를 먼저 실행하라고 안내하고 종료한다
- 포함/제외 기준을 통과한 질문만 생성한다 — 수량 상한선 없음, 억지로 채우지 않음
- 답변란에 추천 옵션이나 제안 답변을 채우지 않는다 — 빈칸으로 남긴다
- 기술 스택 관련 질문은 기능적 명확성을 차단하지 않는 한 피한다
- `clarification.md`가 이미 존재하는 경우, 기존 파일을 덮어쓰기 전에 사용자에게 확인한다
- 스펙 파일은 읽기만 하고 수정하지 않는다 (수정은 별도 단계에서 담당)
