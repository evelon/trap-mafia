# Clarify-Answer — 명확화 질문 답변 및 스펙 반영 스킬

`/clarify-answer` 뒤에 전달된 입력으로 대상 피처의 `clarification.md`를 찾아 답변 프로세스를 진행한다.

## 실행 흐름

### 1. 파일 탐색

`/clarify-answer` 뒤의 인자를 파싱하여 `specs/` 하위에서 대상 파일들을 찾는다.

- 숫자만 전달된 경우 (예: `02`): `specs/` 에서 `02_` 로 시작하는 디렉토리를 찾는다
- 디렉토리명이 전달된 경우: 해당 디렉토리를 직접 사용
- `clarification.md`가 없으면: "`/clarify`를 먼저 실행하여 명확화 질문을 생성하세요." 출력 후 종료
- `spec.md`가 없으면: "`/specify`를 먼저 실행하여 스펙을 생성하세요." 출력 후 종료
- 모든 질문에 이미 답변이 채워져 있으면: "모든 질문에 이미 답변이 완료되어 있습니다." 출력 후 종료

### 2. 질문별 답변 수집

`clarification.md`에서 **답변**이 비어있는 질문을 순서대로 처리한다. 각 질문에 대해:

#### 추천 옵션 제시

질문의 카테고리와 맥락을 분석하여 추천 옵션을 제시한다. 가장 적합한 옵션을 판단하는 기준:

- 스펙의 맥락과 프로젝트 타입에 대한 베스트 프랙티스
- 리스크 감소 (보안, 성능, 유지보수성)
- 스펙에 명시된 목표와의 정합성

**선택지가 있는 질문** (2-5개의 명확한 옵션이 존재할 때):

```
## Q[N]: [질문 제목]

**맥락**: [clarification.md에서 가져온 맥락]

**질문**: [clarification.md에서 가져온 질문]

**추천**: 옵션 A - [추천 이유]

| 옵션 | 설명 |
|------|------|
| A    | [첫 번째 선택지] |
| B    | [두 번째 선택지] |
| C    | [세 번째 선택지] |
| 직접 입력 | 짧은 답변 (5단어 이내) |

> "yes" 또는 "추천"으로 추천 옵션 수락, 옵션 문자(A/B/C) 입력, 또는 직접 답변
```

**자유 답변 질문** (선택지가 의미 없는 경우):

```
**제안**: [제안 답변] - [근거]

> "yes" 또는 "제안"으로 수락, 또는 직접 답변 (5단어 이내)
```

#### 답변 처리

- "yes", "추천", "제안" → 제시한 추천/제안 답변을 사용
- 옵션 문자 (A, B, C 등) → 해당 옵션을 답변으로 사용
- 직접 입력 → 5단어 이내 확인, 모호하면 한 번 재확인 (같은 질문 내에서)
- "skip" → 해당 질문을 건너뛰고 답변란을 비워둠

**한 번에 하나씩** 질문을 제시한다. 다음 질문은 현재 질문의 답변이 확정된 후에만 진행한다.

### 3. clarification.md 업데이트

답변이 확정될 때마다 즉시 `clarification.md`의 해당 질문의 **답변** 란을 채운다.

### 4. spec.md 반영

각 답변 확정 직후, `spec.md`에 다음 두 가지를 반영한다:

#### 4-1. Clarifications 섹션 기록

- `spec.md`에 `## Clarifications` 섹션이 없으면, `## 범위 외` 섹션 바로 앞에 생성한다
- 그 아래 `### Session YYYY-MM-DD` 소제목을 만든다 (오늘 날짜, 같은 날짜면 기존 세션에 추가)
- 답변을 bullet으로 기록: `- Q: <질문> → A: <최종 답변>`

#### 4-2. 스펙 본문 수정

답변 내용에 따라 스펙의 해당 섹션을 직접 수정한다:

| 답변 카테고리           | 반영 위치                                             |
| ----------------------- | ----------------------------------------------------- |
| 기능 범위 & 행동        | 기능 요구사항 섹션에 항목 추가/수정                   |
| 도메인 & 데이터 모델    | 핵심 엔티티 섹션에 필드/관계/제약 추가                |
| 인터랙션 & UX 흐름      | 사용자 스토리 또는 인수 시나리오에 반영               |
| 비기능 품질 속성        | 비기능 요구사항 섹션에 측정 가능한 기준 추가          |
| 엣지 케이스 & 장애 처리 | 엣지 케이스 섹션에 항목 추가                          |
| 통합 & 외부 의존성      | 가정사항 또는 요구사항에 반영                         |
| 제약조건 & 트레이드오프 | 가정사항에 반영                                       |
| 용어 & 일관성           | 스펙 전체에서 용어 통일                               |
| 완료 신호               | 인수 시나리오의 Then 절 구체화                        |
| 기타                    | `[확인 필요]` 마커 제거 및 해당 위치에 확정 내용 삽입 |

반영 원칙:

- 모호했던 기존 문장은 **교체**한다 (중복/모순 남기지 않음)
- `[확인 필요: ...]` 마커가 해당 질문과 관련되면 제거하고 확정 내용으로 대체
- 기존 섹션의 순서와 헤딩 계층을 유지한다
- 각 삽입은 최소한으로, 테스트 가능하게 작성한다

답변 하나를 반영할 때마다 **즉시 파일을 저장**한다.

### 5. 완료 보고

모든 질문 처리 후 다음을 출력한다:

- 답변된 질문 수 / 전체 질문 수
- 건너뛴 질문이 있으면 해당 목록
- `spec.md`에서 수정된 섹션 이름 목록
- `clarification.md` 및 `spec.md` 파일 경로

### 6. clarification.md 정리

완료 보고 후, `clarification.md` 파일 삭제 여부를 묻는다:

```
`clarification.md`의 내용이 `spec.md`에 반영되었습니다.
이 파일은 더 이상 필요하지 않습니다. 삭제할까요? (yes/no)
```

- "yes" → 파일을 삭제하고 "삭제되었습니다." 출력
- "no" 또는 그 외 → 파일을 유지하고 종료

## 행동 규칙

- `clarification.md`가 없으면 `/clarify`를 먼저 실행하라고 안내하고 종료한다
- 질문은 **한 번에 하나씩** 제시한다. 미래 질문을 미리 보여주지 않는다
- 사용자가 "done", "stop", "그만" 등을 입력하면 즉시 중단하고 완료 보고로 넘어간다
- 답변 모호 시 재확인은 같은 질문 내에서 **1회**만 허용한다
- `spec.md` 수정 시 관련 없는 섹션은 건드리지 않는다
- 새로 추가하는 헤딩은 `## Clarifications`와 `### Session YYYY-MM-DD`만 허용한다
