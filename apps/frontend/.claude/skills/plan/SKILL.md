# Plan — 프론트엔드 구현 계획 수립 스킬

`/plan` 뒤에 전달된 입력으로 대상 피처 스펙을 찾아, 코드베이스를 탐색하고, 구현 계획을 수립한다.

## 실행 흐름

### 1. 스펙 파일 탐색

`/plan` 뒤의 인자를 파싱하여 `specs/` 하위에서 대상 스펙을 찾는다.

- 숫자만 전달된 경우 (예: `02`): `specs/` 에서 `02_`로 시작하는 디렉토리를 찾는다
- 디렉토리명이 전달된 경우 (예: `02_room-and-lobby-api-mvp`): 해당 디렉토리를 직접 사용
- 매칭되는 디렉토리가 없으면: "해당 번호/이름의 스펙 디렉토리를 찾을 수 없습니다." 출력 후 종료
- `spec.md` 파일이 없으면: "`/specify`를 먼저 실행하여 스펙을 생성하세요." 출력 후 종료

### 2. 백엔드 API 매핑

`docs/rules/api-integration.md`를 읽고 이 프로젝트의 API 연동 패턴과 생성 파일 역할을 파악한다.

스펙에서 기능 요구사항을 추출한 뒤, 아래 파일을 탐색해 각 요구사항에 매핑되는 API를 찾는다.

**탐색 대상 파일**
- `src/client/gen/@tanstack/react-query.gen.ts` — queryOptions/mutation 팩토리
- `src/client/gen/sdk.gen.ts` — 저수준 API 함수 (SSE 포함)
- `src/client/gen/types.gen.ts` — 요청/응답 타입

**탐색 우선순위**
- REST: react-query.gen.ts의 팩토리 → sdk.gen.ts의 API 함수 → types.gen.ts로 타입 확인
- SSE: sdk.gen.ts 탐색

결과는 `templates/api-mapping.md` 형식을 참고해 `{스펙 디렉토리}/plans/api-mapping.md`에 저장한다.

### 3. 구현 계획 작성

`{스펙 디렉토리}/plans/plan.md`를 `templates/plan.md` 템플릿 기반으로 작성한다.

#### 작성 내용

1. **요약**: 핵심 요구사항 + 기술적 접근 방식 (리서치 결과 반영)
2. **프로젝트 구조**: 실제 생성/수정할 파일 목록과 디렉토리 구조
3. **설계 결정**: 스펙에서 명확하지 않거나 여러 방법이 가능하여 판단한 항목과 근거 기록

#### 작성 원칙

- 템플릿에서 해당 프로젝트에 맞지 않는 섹션은 제거한다
- 플레이스홀더를 실제 값으로 채운다
- 파일 경로는 절대 경로가 아닌 프로젝트 루트 기준 상대 경로로 기록한다

### 4. 완료 보고

모든 산출물 생성 후 다음을 보고한다:

- 리서치에서 발견한 주요 사항 요약

## 행동 규칙

- **스펙 파일은 읽기만 하고 수정하지 않는다**
- 기존 파일이 있으면 덮어쓰기 전에 사용자에게 확인한다
- 산출물은 **한국어**로 작성한다
- `{스펙 디렉토리}/plans/` 디렉토리가 없다면 생성한다
