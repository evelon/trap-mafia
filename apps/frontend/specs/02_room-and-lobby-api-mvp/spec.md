# 피처 스펙: 방/로비 API 연동 및 라우팅 (MVP)

## 사용자 시나리오 및 테스트

### 사용자 스토리 1 - 로비에서 방 참가

로그인을 완료한 유저는 로비 화면(`/rooms`)으로 이동한다. 로비에는 "방 참가" 버튼이 하나 있다. 유저가 이 버튼을 누르면 고정된 하나의 게임 방에 입장하고, 게임 방 화면으로 이동한다.

> MVP에서는 방이 하나뿐이므로 방 목록이나 방 선택 UI는 없다. 이후 버전에서 방 목록이 추가될 예정이다.

**인수 시나리오**:

1. **Given** 로그인이 완료된 유저가 로비 화면에 있을 때, **When** "방 참가" 버튼을 누르면, **Then** 고정된 방에 입장 요청이 전송되고 성공 시 게임 방 화면(`/rooms/current`)으로 이동한다
2. **Given** 로그인이 완료된 유저가 로비 화면에 있을 때, **When** "방 참가" 버튼을 눌렀으나 입장에 실패하면 (방이 꽉 찬 경우 등), **Then** 에러 메시지가 표시되고 로비 화면에 머문다

---

### 사용자 스토리 2 - 게임 방 실시간 정보 구독

유저가 게임 방에 입장하면 즉시 방 상태에 대한 실시간 연결이 수립된다. 이후 방 상태가 변경될 때마다 (다른 유저 입장/퇴장, 게임 시작 등) 최신 정보가 자동으로 반영된다.

> 어떤 정보를 화면에 어떻게 보여줄지는 이 스펙의 범위 밖이다. 이 스펙에서는 실시간 연결 수립과 데이터 수신까지만 다룬다.

**인수 시나리오**:

1. **Given** 유저가 게임 방에 입장했을 때, **When** 방 화면이 로드되면, **Then** 방 상태에 대한 실시간 연결(SSE)이 자동으로 수립되고 현재 방 정보를 수신한다
2. **Given** 실시간 연결이 수립된 상태에서, **When** 다른 유저가 입장하거나 퇴장하면, **Then** 변경된 방 상태가 자동으로 수신된다
3. **Given** 실시간 연결이 끊어졌을 때, **When** 네트워크가 복구되면, **Then** 자동으로 재연결을 시도한다

---

### 사용자 스토리 3 - 방 퇴장

게임 방에 있는 유저는 자발적으로 방을 나갈 수 있다. 퇴장하면 로비 화면으로 돌아간다.

**인수 시나리오**:

1. **Given** 유저가 게임 방에 있을 때, **When** 방 나가기를 실행하면, **Then** 퇴장 요청이 전송되고 성공 시 로비 화면(`/rooms`)으로 이동한다
2. **Given** 유저가 게임 방에 있을 때, **When** 브라우저를 닫거나 새로고침하면, **Then** 서버 측에서 유저 이탈을 감지한다 (SSE 연결 끊김)

---

### 사용자 스토리 4 - 라우팅 및 접근 제어

유저의 현재 상태(로그인 여부, 방 참가 여부)에 따라 적절한 화면으로 안내한다. 방 참가 여부는 `/me` API 응답의 `current_case_id` 필드(uuid|null)로 판단한다.

**인수 시나리오**:

1. **Given** 로그인하지 않은 유저가, **When** `/rooms` 또는 `/rooms/current`에 접근하면, **Then** 로그인 화면으로 리다이렉트된다
2. **Given** 로그인했지만 방에 참가하지 않은 유저가, **When** `/rooms/current`에 접근하면, **Then** 로비 화면(`/rooms`)으로 리다이렉트된다
3. **Given** 이미 방에 참가 중인 유저가, **When** `/rooms`(로비)에 접근하면, **Then** 게임 방 화면(`/rooms/current`)으로 리다이렉트된다

---

### 엣지 케이스

- 이미 방에 참가 중인 유저가 다시 "방 참가"를 시도하면 어떻게 되는가? → 이미 참가 중인 방으로 이동
- 방에 입장했는데 SSE 연결이 즉시 실패하면? → 재연결 시도, 지속 실패 시 에러 상태 표시
- MVP에서는 호스트 개념이 없으므로 호스트 이탈 시나리오는 해당 없음. 방은 고정이므로 마지막 유저가 나가도 해산되지 않음

## 요구사항

### 기능 요구사항

- **FR-001**: 로비 화면에서 고정된 방에 참가할 수 있어야 한다 (MVP: 방 ID 하드코딩)
- **FR-002**: 방 입장 성공 시 게임 방 화면으로 자동 이동해야 한다
- **FR-003**: 게임 방 화면 진입 시 SSE를 통한 실시간 방 상태 구독이 자동으로 수립되어야 한다
- **FR-004**: SSE 연결이 끊어지면 자동 재연결을 시도해야 한다 (재연결 정책의 구체적 수치는 구현 재량)
- **FR-005**: 유저는 게임 방에서 자발적으로 퇴장할 수 있어야 한다
- **FR-006**: 유저의 상태(로그인/방 참가 여부)에 따라 적절한 화면으로 리다이렉트해야 한다

### 핵심 엔티티

- **Room (방)**: 유저들이 모여 게임을 진행하는 공간. MVP에서는 고정된 하나의 방만 존재하며 해산되지 않는다. 호스트 개념 없이 참가자 목록만 포함한다.
- **RoomState (방 상태)**: SSE를 통해 실시간으로 전달되는 방의 현재 스냅샷. 참가자 변동, 게임 시작 등의 변화를 반영한다.

## 가정사항

- MVP에서는 방 ID가 하드코딩되며, 방 생성/목록 조회 기능은 없다
- 유저는 동시에 하나의 방에만 참가할 수 있다 (API 경로가 `/rooms/current`인 점에서 유추)
- SSE 연결은 방 화면에 있는 동안만 유지하며, 화면 이탈 시 연결을 종료한다
- 로그인 후 최초 리다이렉트는 auth 스펙에서 이미 `/rooms`로 처리되고 있다

## Clarifications

### Session 2026-03-21

- Q: 호스트가 방을 떠나면 어떻게 되는가? → A: MVP에서는 호스트 개념이 없다. 모든 유저가 호스트 권한(강제 퇴장 등)을 가진다. 방은 1개로 고정이므로 마지막 유저가 퇴장해도 방은 해산되지 않는다.
- Q: SSE 재연결 "지속 실패" 판정 기준은? → A: 프론트엔드 구현 재량에 맡긴다. MVP에서는 구체적 수치를 스펙으로 규정하지 않는다.
- Q: 라우팅 가드에서 방 참가 여부 판단 방법은? → A: `/me` API 응답의 `current_case_id` 필드(uuid|null)로 판단한다.

## 범위 외

- 게임 방 내부에 표시할 정보의 구체적 UI/UX (별도 스펙에서 다룸)
- 방 생성, 방 목록 조회 (MVP 이후)
- 게임 시작(`case_start`) 이후의 게임 진행 흐름
- 채팅, 음성 등 방 내 커뮤니케이션 기능
- 유저 강제 퇴장(kick): 참가자 목록 UI가 없으므로 이 스펙에서는 다루지 않음 (다른 유저를 대상으로 하는 화면 자체가 범위 외)
